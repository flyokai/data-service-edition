# CLAUDE.md

This is **`flyokai/data-service-edition`** — a project template (used via `composer create-project`) for socket-based async data services on the Flyokai framework.

It includes **everything in [`flyokai/webapi-edition`](https://github.com/flyokai/webapi-edition)** plus the data-service stack: `flyokai/data-service`, `flyokai/data-service-message`, `flyokai/amp-channel-dispatcher`.

## Project shape after install

```
.                        # this template — cloned via composer create-project
├── composer.json        # frozen dependency set
├── composer.lock        # committed; see README.md "composer.lock is committed"
├── scripts/sync-agent-assets.php
├── README.md, CLAUDE.md, .gitignore
├── vendor/              # populated by composer install
├── bin/                 # generated by `vendor/bin/flyok-setup install` (DeployAssets step)
├── storage/             # generated; holds flyok.config.json, certs, oauth keys, …
├── .agents/{commands,skills}/   # symlinked from vendor/*/.agents/* by sync-agent-assets.php
└── .claude/{commands,skills}/   # same — synced for Claude Code agents
```

## Commands

```bash
# Install (first time)
vendor/bin/flyok-setup install --db-host=… --db-user=… --db-pass=… --db-name=… --base-url=…

# Subsequent
php bin/flyok-setup upgrade
php bin/flyok-console ...
php bin/flyok-cluster {start|stop|restart}
```

## Data-service flow

```
Client → Socket → OAuth Handshake → StreamChannel → Dispatcher → Router → Middleware → Handler → Response
```

The first message on every connection MUST be `AccessToken`. After that, requests are routed by class name; middleware is topologically sorted and applied at `Router::onStart()`.

## Agent guidance

- For framework architecture, read `vendor/flyokai/flyokai/README.md` and `vendor/flyokai/flyokai/AGENTS.md`.
- For data-service internals, read `vendor/flyokai/data-service/README.md` and `AGENTS.md`.
- For DI internals, read `vendor/flyokai/flyokai/docs/amphp-injector-flyokai.md` and `vendor/flyokai/amphp-injector/README.md`.
- The [`new-request-handler`](.agents/skills/flyokai-new-request-handler.md) skill scaffolds a request/response/handler triple plus a `diconfig_worker.php` registration on `dataRouterBuilder:default`.
- Routes must be registered before `Router::onStart()`. Adding a route to a started router throws `Error`.
- `WeakReference` is used heavily in `Context` — handlers should null-check `context->dispatcher()` before calling it.
- Messages are serialized via `serialize()`/`unserialize()`. Custom DTOs must support that.

## Conventions

- All flyokai/* dependencies are pinned to `dev-dev`. The `main` branch is stable; `dev` is active.
- The `composer.lock` is committed. **Do not** silently regenerate it during normal work — only as part of a deliberate "bump dependencies" commit, with a smoke test.
- The `bin/` and `storage/` directories are generated, not authored. Don't commit them.
- The agent-asset symlinks under `.claude/` and `.agents/` are regenerated by `composer run sync-agent-assets`.

## Customising

A new project derived from this template typically:

1. Adds its own package (e.g. `acme/data-service-module`) under a `vendor/`-namespaced repo and requires it.
2. Defines its own `Request\…` / `Response\…` DTOs implementing `Flyokai\AmpChannelDispatcher\{Request,Response}`.
3. Registers handlers in `diconfig_worker.php` on `dataRouterBuilder:default`.
4. Tags Solid DTOs with `#[Table]` so the schema framework reconciles MySQL on `setup:upgrade`.

Use the skills in `.agents/skills/` to scaffold each step rather than copy-pasting boilerplate.
